The shared problem: student life

Everyone builds in the same problem space — helping a student organize their study life — but each team builds different features. Same domain, different solutions. (So you can't copy, and the teacher's examples still apply to your team.)

You live this pain: classes, deadlines, group work, club activities, scattered notes, messy communication. That's why you can write great requirements for it.

Two layers (so you don't drown)

Layer A — everyone builds this core (small, doable):

  1. Add classes / deadlines / tasks (with due date)
  2. Remind me before a deadline
  3. See my week at a glance
  4. One AI-assisted feature, e.g. type "group SE meeting Tuesday 3pm" → AI turns it into a calendar event (one API call — no model training)

Layer B — your team picks 1–2 of your own:

Theme Example features
Group work shared task board · who-does-what · group deadline tracker
Notes notes per subject · AI note summary · flashcards
Planning study-time suggester · exam countdown · weekly goals
Activities club/event finder · sign-up · attendance check-in
Communication group announcements · reminders to teammates · meeting poll
Focus pomodoro timer + study stats
Matching find a study buddy by course/time

Want a different idea? Some teams may pick another real-world topic — ask the instructor.

The teaching example: StudyMate

In class the instructor demos StudyMate (a student study assistant) each week, then you apply the same step to your own project. Full walkthrough: the “StudyMate” walkthrough handed out in class.

One feature, all the way down (the model to copy)

Layer StudyMate example
Requirement (what) FR-3: remind 1 day before a deadline; not for done tasks
Design (how) a daily scheduler + a Task entity
Code tasks_due_for_reminder(...) + a commit that says (FR-3)
Test tests that match the acceptance criteria

Do this for your features and the Golden Thread takes care of itself.

Security & ethics (you'll be graded on basic awareness)

  • A schedule is personal → a student should see only their own tasks.
  • Don't put secrets/passwords in your repo.
  • AI features can be wrong (it may read "Tuesday" as "Wednesday") → a human checks; get consent; don't expose group chats.

Tips

  • Start the SRS in Week 3 — it drives everything.
  • Use Git from Week 6; everyone commits (it's graded).
  • Keep your AI Usage Log as you go, not at the end.
  • Keep the MVP small and finished, then add one team feature.
