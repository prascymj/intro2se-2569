Concept 1 / 5

What a software process is

A process is the ordered set of activities to produce software. The four fundamentals are specification, design & implementation, validation, and evolution. A “model” is one way of arranging them.

Example
Even a 5-person student project follows them: agree what to build → build it → test it → change it.

Concept 2 / 5

Plan-driven vs iterative

Plan-driven (waterfall): do all of one stage before the next — simple but risky, because you only see working software at the end. Iterative/incremental: build a small working slice, then add more — you learn early and adjust.

Example
Waterfall: write ALL requirements, then design everything, then code for months. Iterative: ship a working reminder in 2 weeks, then add reports.

Concept 3 / 5

Agile basics (Scrum & Kanban)

Agile delivers in short cycles with constant feedback. Scrum works in fixed “sprints” with roles and a backlog; Kanban uses a board (To-do / Doing / Done) to limit work in progress. Great for small student teams.

Example
Your team runs 2-week sprints: each sprint ends with something that actually works and a quick review.
Intro2SE · Ch2
Concept 4 / 5

MVP — Minimum Viable Product

The smallest version that delivers real value and can be tested with users. Build the core first; resist adding everything at once.

Example
StudyMate MVP = add a task + get a reminder. NOT MVP = AI chatbot + social feed + dark mode on day one.

Concept 5 / 5

Choosing a process

Pick based on clarity and size: stable, well-understood needs → more plan-driven; changing or unclear needs (most student projects) → iterative/Agile.

Example
Your project is new and you'll learn as you go → choose incremental/Agile, and say WHY in your charter.
